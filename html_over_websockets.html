WebSockets, what and why ?

The WebSocket protocol is built on top of TCP, and uses a handshake process to establish the connection, followed by a message-based communication protocol. This allows for bidirectional, low-latency communication between the client and the server.

So with WebSockets, I could write server-side Python code that would send HTML over the WebSocket connection to be dynamically displayed in the user's browser. Because logic lives on the server and the browser deals just with the HTML , I can stick to Python, eliminating the need to learn a separate front-end framework or language.

The benefits of WebSockets over HTTP requests...

  • Real-time communication : Front-end and back-end communicate in real-time,

eliminating the need for the HTTP request/response cycle.

  • Low-latency: WebSockets have lower latency than HTTP requests due to the

persistent connection between back-end and front-end.

  • Reduced server load: WebSockets reduce server load since they require fewer

connections and less data transfer than HTTP requests (more on that later).

  • Full-duplex communication: With HTTP requests, the client can only request data

from the server, and the server can only respond to these request. With WebSockets, client and server can send and receive data at the same time.

  • Scalability: Websockets require fewer connections and can handle a higher

volume of data with lower latency due to the fact that connections are kept open. Additionally, WebSocket headers are much smaller and simpler than the typical HTTP header.

  • SEO: When the server generates a complete HTML response, including content and

metadata, this allows search engines to crawl the entire page.

  • Javascript: Developers can create a Single Page Application with hardly any

Javascript.

... and some drawbacks.

Because of persistent connection between client and server, WebSockets can be vulnerable to attacks such as denial-of-service, state management can be more complex than with HTTP requests. Additionally, more server resources are required to handle many concurrent connections, which can impact scalability. But i suppose all of this can be handled with proper implementation :P

Another drawback is the ecosystem, there's a few comprehensive frameworks, documentation, and tutorials to aid developers in building and maintaining WebSocket-based applications.

What can i build ?

HTML over the wire can be useful in all sort of real-time applications, for example :

  • Chat
  • Multiplayer games
  • Stock trading platforms
  • Collaborative editing tools
  • Analytics dashboards

I'm ready to dive in !

  • Django Channels: a Django library that provides WebSocket support and allows

you to build real-time applications.

  • Phoenix LiveView: Phoenix is a web framework written in Elixir. Chris McCord,

the creator of Phoenix presented HTML over WebSockets with Liveview by building a Twitter clone in 15 minutes.

  • Hotwire: Stands for HTML over the wire. It's the default frontend solution in

Rails, and very popular in the Laravel community, since it's not tied to any framework.
